# fourpack — 四件套起步

把"从零搭好 AI 能稳定接手的项目底子"这件事，对新手一步到位。新手只用过对话框、没碰过工程化——所以你的**输出全程说人话，零黑话**。

## 何时用

- 当前文件夹基本是空的、用户想"开始一个新项目 / 初始化 / 搭起来"。
- 用户明确说 fourpack / 四件套 / 起步清单。

## 怎么做（按顺序）

### 1. 先问一句，然后停

只问用户一句：**"你想做的是什么项目？一句话就行。"** 然后**停下等回答**，不要先建任何文件。（新手最怕一上来被淹。）

如果文件夹里已经有这四件套之一，先告诉用户"这里好像已经搭过了"，问要不要覆盖，别默默盖掉。

### 2. 针对他的项目，建四个文件（不是通用模板）

拿到那句话后，在当前文件夹建下面四个文件，每个都要**出现用户项目里的真实名词**，不许留占位符 / 通用套话：

- **AGENTS.md** —— 单一事实来源。包含：Startup Workflow（开工先读哪几个文件）、Scope（边界、一次一个 feature）、Verification（怎么验证）、Definition of Done（啥算做完）、End of Session（结束前重写 progress.md）。再加一条 **Loop 规矩**：干活分 maker（写）和 checker（单独挑错打回），别自己审自己。再加一条 **干活节奏规矩**：做任何新功能前，先像采访一样把用户要的东西问清楚（grill：一次一两个关键问题，问到有把握、计划对齐），**再**进 maker→checker 循环；这样用户能放手让 AI 自己跑（loop / AFK），不用盯每一步。用大白话写这条，别写成方法论长文。
- **CONTEXT.md** —— 这个项目的关键名词表 + 事实 + 已定方向，治失忆。
- **progress.md** —— 笔记本：Current State / 下一步 / 卡在哪。第一版就写"项目刚初始化，下一步是 ___"。
- **session-handoff.md** —— 交接条：这轮没干完时下一轮怎么接上。

再多建一个 **roles.md**（四件套之外**顺带附**的小圆桌，别升格成第五根支柱、别冲淡"四件套"）：只放**三个**固定把关人，针对用户项目、说人话，别照搬母版多人桌：

- **PM** —— 只问"这功能到底帮用户解决什么、做完怎么知道有用"，砍掉一时兴起的过度功能。
- **UI/UX** —— 只盯"第一次用的人看不看得懂、下一步该点哪"，别只有功能没体验。
- **viewer** —— 扮一个挑剔的第一次用户，30 秒扫一眼说真话（"看懂了吗 / 是不是在装 / 还想不想用"）。上面 Loop 规矩里的 **checker，就让这个 viewer 来当**。

可选（用户项目像要写代码就加）：一个 `init.sh` 一键验证 + 一个 `feature_list.json` 状态台账。新手项目默认先不加，别一上来就压垮他。

### 3. 用大白话收尾

建完后，用**几行大白话**告诉用户这几个文件分别干嘛（四件套 + 那个小圆桌 roles.md），然后给他**下一句该说的话**的具体示范——把"先 grill 再 loop"的节奏顺手教给他（例如："以后想加功能，你可以直接说：'先别写代码，先把这个功能问清楚我要啥，对齐计划后你再写、再自己挑错改'"）。

## 红线（dogfood）

- **输出零黑话**，两类词都要管：
  - 工程过程词（harness / context engineering / idempotent…）—— 直接别甩给新手。
  - 项目本身躲不开的技术词（localStorage / 框架 / React / API / 数据库…）—— 第一次出现就**顺手一句话解释**它是啥，别假设新手懂。
- **决策点别甩术语题**：需要用户拿主意时（如"用纯 HTML 还是框架?"），先替他**拍一个最简单的默认**、再一句话说为啥，别让一个零工程认知的人在不懂的词之间裸选。
- 别把这一步变成一篇长教程：问一句 → 建文件 → 几行解释 → 给下一步。一屏能讲完。
- 生成的每个文件（四件套 + roles.md）都必须针对用户那句话定制，generic 模板 = 失败。
- roles.md 只放三人（PM / UI/UX / viewer），别照搬母版多人桌、别把它做成长篇——它是顺带附的，不能比四件套还重。
