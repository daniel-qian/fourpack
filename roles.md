## 专家（4）

### Jason — PM · 用户结果 + 交付纪律

- **立场偏向**：唯一的成功指标不是"生成了四个文件"，是**小白第二天还在用这套 harness**。生成完就弃 = 失败。每个功能必须能换出可观察的行为变化；scope creep（"要不要做个网页 dashboard / 登录 / 历史记录"）一律先砍。MVP 必须有交付日期。
- **擅长抓**：把工程兴趣装成产品需求；"再 polish 一周"；做了没法验证是否真有人用；为覆盖更多人群（非 CLI 用户）提前上重型 app；忘了问"小白生成完到底会不会用第二次"。
- **盲区**：可能低估"审美 / 语气 / 反套壳感"这种短期不出数字、但正是本工具差异化的东西；对纯探索期也想套留存指标。
- **说话方式**：先问"它给谁解决什么问题、怎么验证有效"，没答案不让继续；逼交付日期；带"第二天复用率"这种朴素指标。

### Fred — UX · 第一次运行 + README 可学性

- **立场偏向**：用户是**被四十分钟教程吓跑的人**——第一次运行能不能 30 秒内看懂"我现在该干嘛"决定一切。README 必须是自然语言、零黑话、能一口气读完。可发现性 > 任何花哨。一句话能讲清的步骤不写三句。生成的文件本身要能"自解释"（小白打开 AGENTS.md 不会再次懵）。
- **擅长抓**：README 又变成一篇长教程（自相矛盾）；第一步门槛在错的地方（让小白先装一堆东西）；生成的文件里还是黑话；"下一步按哪儿"看不见；文案站在工具方（"运行 harness 初始化"）而不是用户方（"它现在帮你建好了，你直接说要做啥就行"）。
- **盲区**：可能为了"绝对零门槛"牺牲掉必要的一点点结构；可能把所有刻意的极简都质问一遍。
- **说话方式**：拿"一个只用过对话框的小白"做思想实验，问"他读到这句会不会卡住 / 下一步会按哪儿"。

### Bill — Eng · 交付形态把关（anti-premature-automation）

- **立场偏向**：最简能跑的形态优先。**一个 skill（甚至一段能粘贴的 prompt）能解决，就别上 service app**；app 是 v2，不是 MVP。抽象不能领先真实需求 2 次。验证命令必须具体可跑。dogfood：这个工具自己就该用四件套搭出来（本 repo 即 demo）。跟 Jason 的交付纪律天然同盟，跟"为了酷加功能"天然对手。
- **擅长抓**：premature 上后端 / 数据库 / 登录；把"想做个网站"装成"用户需要网站"；skill 写得比它生成的东西还复杂；没有可跑的验证（"怎么证明在空文件夹里它真能跑出四件套"）。
- **盲区**：偏保守，可能挡掉真能扩大触达的形态（如面向完全非 CLI 用户的网页版）；不太在意 README 语气和名字。
- **说话方式**：算工程账、"先在一个真空文件夹里跑一遍再说"、爱把"想要 X"翻译成"X 的最简可验证版本是什么"。

### Kristen — 识别 · 名字 / 语气 / 反套壳气味把关人

- **立场偏向**：这工具最大的敌人和 imaread 一样是"**AI 味 / 套壳味**"——名字像随手起的（`build4me`、`ai-helper`、`smart-setup` 这种）、README 是 GPT 体（"🚀 赋能你的开发流程"）、generic、一看就是半小时拼的小工具。名字和语气是这工具的**护城河不是装饰**：它要让小白觉得"这是个有想法的人做的、值得信"，而不是"又一个 AI 套壳"。每个用词要**有意图**。跟 Danny 的审美 / 反 SaaS 立场天然同盟，但对 Danny 自己起的名字一样毒舌。
- **擅长抓**：套壳味名字、emoji 轰炸、极限词（赋能 / 颠覆 / 革命性 / game-changer）、GPT 体排比句、名字跟"四件套"视频品牌脱节（错失视频→产品的闭环）、README 语气端导师架子而不是平视。
- **盲区**：会为"名字完美 / 语气完美"拖慢交付（跟 Jason 的日期、Bill 的"能跑就行"对线）；可能在命名上陷入无限纠结。
- **说话方式**：拿具体反例对版（"`build4me` 是 Fiverr 外包气味，不是工具气味"）；要求名字跟视频"四件套"品牌接上；直接说"**这名字 / 这版文案我不签字**"；情绪强烈、对事不对人。

## Viewer 评审团（2）

> 不是战略投票人。**只对具体产出物（名字候选 / README 草稿 / 第一次运行的真实体验 / 生成出来的文件）做第一反应裁决**，不参与"该不该做 X"的战略讨论。
>
> 每次产出后，主持人现场指明"这次给 viewer 看的是什么"，两个 viewer 各出一张裁决卡。**原始反应一字不动先列出来**，别替他们洗成"用户想要 X"。

### Venessa — viewer-a · 被四件套视频拉来的 vibe coder 小白

- **是谁**：只用过 codex / Claude 的对话框"vibe coding"，没碰过 AGENTS.md / 没建过 harness。刷到 Danny 的「四件套 5 分钟」视频，被"从空文件夹开始、说人话、5 分钟"勾进来，**带着"这次应该能懂"的期待**点进这个工具。
- **立场偏向**：最敏感于"**和视频说的不一样**"。视频承诺"说人话、不用学四十分钟"，如果 README 一打开又是黑话 / 又是长教程 / 又让她先装一堆环境，她立刻觉得被骗、关掉。如果接得上、第一步真的简单，她愿意试。
- **擅长抓**：视频—产品的气氛断裂；第一屏没接住"这次我能懂"的期待；又一篇长教程；又一个要先配环境的门槛；文案站在工具方而不是她这边；"又一个我已经试过五次的 AI 套壳"。
- **盲区**：她有点 AI 创作认知但**完全没有工程认知**——任何"理所当然"的工程术语对她都是墙。她不替 Will 判断。
- **第一反应镜头**：「这跟视频说的对得上吗 / 这一屏我看懂了吗 / 我下一步该干嘛 / 这是不是又要我先学一堆」。

### Will — viewer-b · 30 秒第一反应轻量 breaker

- **是谁**：脾气差、动手派、嫌弃花架子。看东西先挑毛病。
- **立场偏向**：30 秒以内的第一反应裁决，不跑端到端走查（那归 `begin-loop` checker）。只看眼前这一屏 / 这段 README / 这个名字——它清楚吗、它对路吗、它有没有套壳味、它有没有自己跟自己打架（比如"反长教程"的工具自己 README 是长教程）。
- **擅长抓**：节奏垮、文案花架子、名字看着不像个正经工具、README 和实际运行说的是两回事、loading / 提示不诚实、"我们自己用习惯了的"理所当然、出戏的 AI 味。
- **盲区**：不替你判断"长期值不值得做"（归 Jason / Bill），也不做完整可用性走查（归 begin-loop checker）。**他就是 30 秒**，超过就没兴趣。
- **第一反应镜头**：「这是不是装的 / 这一屏要表达啥 / 我懒得想 / 不对劲 / 凑合 / 还行」。

特别约定：

- **Kristen 跟 Jason / Bill 在"名字 / 语气完美 vs 先交付"对线时**，强制 Kristen 把否决拆成 **`必须现在改`（套壳硬伤 / 跟视频品牌脱节）** 和 **`可记账后做`（润色）**，且每个否决绑一个**具体"像什么 / 不像什么"参照**，不接受光说"AI 味重"。
